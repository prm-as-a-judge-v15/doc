User Manual

PRM-as-a-Judge v1.5

Turn rollout videos into dense progress curves, then compute trajectory quality, near-success, and robustness diagnostics. Most users only need a JSONL manifest and one command.

Overview

PRM-as-a-Judge converts robot rollout videos into dense progress curves and higher-level trajectory diagnostics. Version 1.5 is organized around a simple workflow: 

JSONL manifest -> PRM adapter -> normalized progress curve -> postprocess -> metrics/report
The main design goal is a low-friction first run: defaults are already selected, so users can get evaluation results with one command. Advanced users can later tune sampling, views, GPUs, smoothing, or PRM backends from this guide.

Quick Start

Install the toolkit from the repository root:

pip install -e .

Run a lightweight recorded-progress demo that does not load a model:

prm-judge eval --prm recorded --manifest eval/examples/manifest_recorded.jsonl

Add --visualize when you want curve plots and a per-case metrics table:

prm-judge eval --prm recorded --manifest eval/examples/manifest_recorded.jsonl --visualize

Run with Robo-Dopamine

MANIFEST=/path/to/cases.jsonl \
PRM_PATH=/path/to/Robo-Dopamine-GRM-8B-Pro \
bash eval/run_eval.sh

The one-command shell runner also exposes common advanced options:

MANIFEST=/path/to/cases.jsonl \
VISUALIZE=1 \
SMOOTHING_WEIGHTS=0.05,0.15,0.6,0.15,0.05 \
bash eval/run_eval.sh

Run with Robometer

By default, the runner starts a local Robometer HTTP server, waits for /health, evaluates cases, and shuts the server down when the run finishes: 

PRM=robometer \
MANIFEST=/path/to/cases.jsonl \
bash eval/run_eval.sh

To reuse an already running Robometer server:

PRM=robometer \
MANIFEST=/path/to/cases.jsonl \
ROBOMETER_SERVER_URL=http://localhost:8000 \
ROBOMETER_AUTO_START=0 \
bash eval/run_eval.sh

Outputs

Outputs are written to eval/results/run_YYMMDD_HHMMSS/ by default:

  • per_case.jsonl: one result object per rollout.
  • summary.csv: task-level v1.5 metrics.
  • leaderboard.csv: model-level summary.
  • report.md: short human-readable run report.
  • discovery_manifest.json: resolved inputs for reproducibility.
  • robometer_server_PORT.log: run-level Robometer server log when auto-start is used.
  • visualizations/curve_metrics.csv: per-case curve metrics when --visualize is enabled.
  • visualizations/case_plots.csv: index of generated per-case curve figures.
  • visualizations/cases/*.png: annotated raw and processed progress curve figures when matplotlib is installed.

You can also generate visualization artifacts after a run has finished:

prm-judge visualize --run-root eval/results/run_YYMMDD_HHMMSS

Manifest Format

Each JSONL line describes one rollout case. Paths may be absolute or relative to the manifest file. UTF-8 files with or without a BOM are accepted.

Single View

{"case_id":"case_001","task_name":"put_can","task":"put the can into the basket","video":"videos/case_001.mp4"}

Multi-View

{"case_id":"case_002","task_name":"put_can","task":"put the can into the basket","videos":{"high":"high.mp4","left":"left.mp4","right":"right.mp4"}}

With Goal Image and Label

{"case_id":"case_003","task_name":"arrange_flowers","task":"arrange the flowers in the vase","videos":{"high":"h.mp4","left":"l.mp4","right":"r.mp4"},"goal_image":"goal.png","label":"success","benchmark":"RoboTwin","model":"pi0"}
Field Description
task Natural-language task description passed to the PRM backend.
task_name Short stable task identifier used for filtering, grouping, output paths, and leaderboard rows. If omitted, it falls back to task.
label success or failure. By default, success-conditioned metrics use MaxP >= --success-threshold; pass --success-source label when benchmark labels should override the progress threshold.
benchmark Grouping key in reports.
model Policy or robot model name.
metadata Free-form JSON object copied into outputs.
progress / progress_path Precomputed progress curve for --prm recorded.

Defaults

The default path is intentionally conservative and ready to run:

  • PRM backend: dopamine
  • Eval mode: backward
  • Frame interval: 72
  • Batch size: 10
  • GPU selection: GPUS=0
  • Outlier removal: local median replacement
  • Smoothing: weighted window [0.1, 0.2, 0.4, 0.2, 0.1]

These defaults provide a one-command evaluation path. Tune them only when you need different speed, sampling density, multi-view handling, goal-image behavior, or PRM output semantics.

Dopamine Modes

Robo-Dopamine supports three progress modes:

  • backward: default. Scores how far the current state is from the goal anchor and is usually the most stable choice for full-trajectory auditing.
  • forward: compares progress from the start toward each current state. Use it when goal evidence is weak but start-to-current progress is clear.
  • incremental: estimates relative progress between adjacent sampled states. It is more sensitive to local noise, so larger frame intervals can help.

Robometer does not use Dopamine's modes. The HTTP adapter sends one selected view to a Robometer eval server and reads outputs_progress.progress_pred as a per-frame progress curve.

Multi-View and Goal Images

For Robo-Dopamine, the preferred input is three synchronized views: high, left, and right. The manifest loader maps them to cam_high, cam_left_wrist, and cam_right_wrist.

If only one video is provided, the Dopamine adapter reuses it for all three camera slots. This is convenient for quick tests, but multi-view inputs usually produce stronger judgments.

For Robometer, the current HTTP adapter evaluates one view. With a multi-view manifest it uses cam_high by default; choose another view with --robometer-view left or --robometer-view right.

Goal images are optional. If goal_image is omitted, the runner uses eval/examples/blank.png when available.

Multi-GPU

Use the GPUS environment variable or --gpus:

GPUS=0,1,2,3 MANIFEST=cases.jsonl PRM_PATH=/path/to/prm bash eval/run_eval.sh

The runner shards cases across GPUs by case index. Each worker loads one PRM instance on its assigned GPU, then shard outputs are merged at the end.

--gpus auto uses nvidia-smi and selects GPUs with low memory usage.

Progress Curve Standard

The metric stack only consumes normalized absolute progress curves:

progress[t] in [0, 1]
  • Absolute [0, 1]: use directly.
  • Absolute [0, 100]: divide by 100 with --normalize 0_100.
  • Relative increments: use --source-type delta; the runner accumulates deltas then clips to [0, 1].
  • Uncalibrated rewards: add adapter-specific normalization before metrics.

Robometer's progress_pred is treated as absolute [0, 1] progress by default. It is not forced to be monotonic because regressions and oscillations are used by CRA, MDD, Rec, and RBS.

Every result stores both progress_raw and progress_processed.

Postprocessing

The default postprocessing pipeline is:

  1. Normalize to [0, 1].
  2. Clip to [0, 1].
  3. Replace isolated local outliers with a local median.
  4. Smooth with a weighted window.
p'_t = 0.1*p_{t-2} + 0.2*p_{t-1} + 0.4*p_t + 0.2*p_{t+1} + 0.1*p_{t+2}

Disable outlier removal or smoothing with:

prm-judge eval --manifest cases.jsonl --outlier-method none --smoothing none

Advanced users can change the smoothing weights from the CLI. The list must have odd length, non-negative values, and a positive sum; it is normalized automatically before use. 

prm-judge eval --manifest cases.jsonl --smoothing-weights 0.05,0.15,0.6,0.15,0.05

For deeper customization, edit eval/prm_judge/curves.py. The relevant extension points are parse_smoothing_weights, weighted_smooth, and PostprocessConfig.smoothing_weights.

Metrics

Case-Level Metrics

  • MaxP: maximum reached progress.
  • M25, M50, M75: milestone indicators.
  • SR: success indicator at the success threshold, default 0.99.
  • PPL: progress-weighted path length anchored at best reached progress.
  • CRA: cumulative regret after progress regressions.
  • Stag: fraction of near-stagnant transitions.
  • MDD: maximum progress drawdown.
  • Rec: recovered progress after the deepest drawdown trough.
  • RBS: robustness score, approximately (Rec + epsilon) / (MDD + epsilon), clipped to 1.

Group-Level v1.5 Metrics

  • MaxP-F: average MaxP among failed trajectories.
  • SQS: success quality score.
  • FNS: failure near-success score.
  • RBS: mean trajectory robustness after the largest drawdown.
  • OAS: overall auditing score combining SQS, FNS, RBS, and SR.

By default, SR_trace is derived from MaxP >= --success-threshold. Use --success-source label only when benchmark labels should define success or failure grouping.

Custom PRM Adapter

Add a backend by subclassing BasePRMAdapter:

from prm_judge.prm.base import BasePRMAdapter
from prm_judge.schema import EvalCase, ProgressTrace

class MyAdapter(BasePRMAdapter):
    def predict(self, case: EvalCase, output_dir):
        scores = run_my_model(case.videos, case.task, case.goal_image)
        return ProgressTrace(case_id=case.case_id, progress=scores, source_scale="0_1")

Then register the backend in eval/prm_judge/cli.py and document the model's output scale and semantics.

The most important adapter decision is output semantics: absolute progress or step-wise deltas, [0, 1], [0, 100], or uncalibrated rewards. Before metrics are computed, everything should become an absolute progress curve in [0, 1].
