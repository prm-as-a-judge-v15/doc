User Guide

PRM-as-a-Judge v1.5

A practical toolkit for robot process auditing. Give it rollout videos and task descriptions; it produces normalized progress curves, case-level diagnostics, task summaries, and leaderboard-ready reports.

With the default settings, users only need to provide a JSONL manifest and run one command to obtain evaluation results. For advanced requirements, this guide explains how to tune sampling, modes, views, GPUs, smoothing, metrics, and PRM backends.

Overview

PRM-as-a-Judge evaluates the process of a robot rollout, not only the terminal outcome. A PRM backend estimates progress over time, then the metric stack turns that curve into interpretable signals about near success, stagnation, drawdown, recovery, and success quality. 

JSONL manifest -> PRM adapter -> normalized progress curve -> postprocess -> metrics/report
The default settings provide a fast and simple path to evaluation results. You can adjust the later stages according to your task, data format, runtime resources, and analysis goals.

Input

Each JSONL line is one rollout case with video paths, task text, optional labels, and metadata.

Judge

Adapters support pair-style PRMs such as Robo-Dopamine and sequence-style PRMs such as RoboMeter.

Report

Outputs include raw and processed curves, per-case metrics, task summaries, leaderboards, and plots.

Quick Start

Install the toolkit from the project repository root:

pip install -e .

1. Create a manifest

Put your rollout paths and task descriptions in a JSONL file. Videos can stay anywhere on disk; paths may be absolute or relative to the manifest. 

{"case_id":"case_001","task_name":"put_can","task":"put the can into the basket","video":"videos/case_001.mp4","model":"pi0"}

2. Run the default evaluation

The default runner uses Robo-Dopamine, backward mode, the default sampling interval, GPU 0, outlier replacement, and weighted smoothing. 

MANIFEST=/path/to/cases.jsonl \
PRM_PATH=/path/to/Robo-Dopamine-GRM-8B-Pro \
bash eval/run_eval.sh

3. Add visualizations

MANIFEST=/path/to/cases.jsonl \
PRM_PATH=/path/to/Robo-Dopamine-GRM-8B-Pro \
VISUALIZE=1 \
bash eval/run_eval.sh

Try without loading a model

Use the recorded backend when you want to verify installation, report generation, and metric computation with precomputed curves. 

prm-judge eval --prm recorded --manifest eval/examples/manifest_recorded.jsonl --visualize

Run with RoboMeter

RoboMeter is a sequence-style backend. The runner can start a local RoboMeter HTTP server, wait for /health, evaluate cases, and shut the server down after the run. 

PRM=robometer \
MANIFEST=/path/to/cases.jsonl \
bash eval/run_eval.sh

Reuse an already running server when you want to manage the model process yourself:

PRM=robometer \
MANIFEST=/path/to/cases.jsonl \
ROBOMETER_SERVER_URL=http://localhost:8000 \
ROBOMETER_AUTO_START=0 \
bash eval/run_eval.sh

Manifest Format

The recommended default input is a manifest file. It uses UTF-8 JSONL, with each line describing one rollout case. Directory-based video input is also supported.

Single View

{"case_id":"case_001","task_name":"put_can","task":"put the can into the basket","video":"videos/case_001.mp4"}

Multi-View

{"case_id":"case_002","task_name":"put_can","task":"put the can into the basket","videos":{"high":"high.mp4","left":"left.mp4","right":"right.mp4"}}

With Goal Image and Benchmark Label

{"case_id":"case_003","task_name":"arrange_flowers","task":"arrange the flowers in the vase","videos":{"high":"h.mp4","left":"l.mp4","right":"r.mp4"},"goal_image":"goal.png","label":"success","benchmark":"RoboTwin","model":"pi0"}

With Precomputed Progress

{"case_id":"case_004","task_name":"open_drawer","task":"open the drawer","video":"drawer.mp4","progress_path":"curves/case_004.json"}
Field Required Description
case_id Recommended Stable case identifier. If omitted, the loader should derive one from the row index or video name.
task Yes Natural-language task description passed to the PRM backend.
video One of video, videos, or progress Single rollout video path. The Dopamine adapter can reuse it for all camera slots.
videos One of video, videos, or progress Object containing view-specific paths, usually high, left, and right.
task_name No Short grouping key used for filtering, output paths, summaries, and leaderboard rows. Falls back to task.
goal_image No Image of the desired final state. Useful when the task text is ambiguous or the target layout matters.
label No success or failure. Use --success-source label when benchmark labels should define grouping.
benchmark No Dataset or benchmark name copied into reports.
model No Policy, robot model, or method name used for leaderboard aggregation.
metadata No Free-form JSON object copied into outputs for reproducibility.
progress / progress_path No Precomputed progress curve for --prm recorded, debugging, or external PRM integration.

Outputs

Outputs are written to eval/results/run_YYMMDD_HHMMSS/ by default.

File Use it for
per_case.jsonl Raw and processed progress curves plus metrics for each rollout.
summary.csv Task-level v1.5 metrics and success/failure grouping.
leaderboard.csv Model-level summary for comparing policies or PRM backends.
report.md Human-readable run report for sharing results.
discovery_manifest.json Resolved input paths, runtime choices, and reproducibility metadata.
visualizations/curve_metrics.csv Per-case curve metrics when --visualize or VISUALIZE=1 is enabled.
visualizations/cases/*.png Raw and processed progress curve plots when matplotlib is installed.

Generate visualizations after a run has finished:

prm-judge visualize --run-root eval/results/run_YYMMDD_HHMMSS

Default Settings

Defaults are selected to make the first run easy and reproducible. Treat this as the baseline configuration before tuning for speed, sampling density, or a different PRM backend.

Setting Default Change when
PRM backend dopamine You want a sequence-style backend such as robometer, or a custom adapter.
Dopamine mode backward Goal evidence is weak, or you need local step-wise progress estimates.
Frame interval 72 You need denser curves for short or fast tasks, or faster inference for long videos.
Batch size 10 You are memory-limited or want higher throughput on larger GPUs.
GPU selection GPUS=0 You have multiple local GPUs and want to shard cases across them.
Outlier removal Local median replacement Your PRM is already stable or you are auditing raw model behavior.
Smoothing [0.1, 0.2, 0.4, 0.2, 0.1] You need a narrower or wider temporal window for very noisy or very fast progress.

Dopamine Modes

Robo-Dopamine is a pair-style PRM. It constructs a progress curve by scoring sampled states with one of three comparison modes.

Mode Best for Notes
backward Default full-trajectory auditing with a clear goal state. Usually the most stable first choice because each sampled state is compared against the goal anchor.
forward Tasks where start-to-current progress is easier to judge than distance-to-goal. Useful when the final state is visually ambiguous or the goal image is unavailable.
incremental Local motion analysis and detecting small step-wise changes. More sensitive to noise and repeated motion; use with care and consider a larger frame interval.

Examples:

MODE=backward MANIFEST=cases.jsonl PRM_PATH=/path/to/prm bash eval/run_eval.sh
MODE=forward MANIFEST=cases.jsonl PRM_PATH=/path/to/prm bash eval/run_eval.sh
MODE=incremental FRAME_INTERVAL=96 MANIFEST=cases.jsonl PRM_PATH=/path/to/prm bash eval/run_eval.sh
Pair-style judging can struggle with repeated or context-dependent motions because a local pair may not reveal whether the policy is making progress or simply moving back and forth. For these cases, compare results with a sequence-style backend such as RoboMeter.

Views and Goal Images

Multi-view and goal-image inputs are optional, but they often improve judgment quality for manipulation tasks where progress is hidden from a single camera.

Recommended view names

  • high: third-person or overhead view. This is the default view for RoboMeter.
  • left: left wrist camera.
  • right: right wrist camera.

For Robo-Dopamine, the manifest loader maps high, left, and right to cam_high, cam_left_wrist, and cam_right_wrist. If only one video is provided, the adapter reuses it for all camera slots so quick tests still work.

For RoboMeter, the current HTTP adapter evaluates one selected view from the manifest. With multi-view input it uses high by default. 

PRM=robometer ROBOMETER_VIEW=left MANIFEST=cases.jsonl bash eval/run_eval.sh

When to add a goal image

  • Use goal_image when the desired final arrangement is specific, such as object layout, color order, or target container.
  • Omit it for simple semantic tasks where the language instruction is enough.
  • If omitted, the runner can fall back to eval/examples/blank.png when the backend requires an image field.

Multi-GPU

The default uses one GPU to keep the command predictable. On larger batches, set GPUS or pass --gpus to shard cases across local devices. 

GPUS=0,1,2,3 MANIFEST=cases.jsonl PRM_PATH=/path/to/prm bash eval/run_eval.sh

Each worker loads one PRM instance on its assigned GPU, processes its shard, and the runner merges shard outputs at the end. This is simple local data parallelism and is usually enough for evaluation batches.

Use automatic selection when you want the runner to choose low-memory GPUs reported by nvidia-smi:

GPUS=auto MANIFEST=cases.jsonl PRM_PATH=/path/to/prm bash eval/run_eval.sh

Progress Standard

The metric stack consumes normalized absolute progress curves. Every PRM adapter should eventually produce: 

progress[t] in [0, 1]
PRM output Convert before metrics
Absolute progress in [0, 1] Use directly, then clip small numerical overflow.
Absolute progress in [0, 100] Divide by 100 with --normalize 0_100.
Step-wise delta progress Use --source-type delta, accumulate over time, then clip to [0, 1].
Uncalibrated reward or similarity Add adapter-specific calibration before metric computation.

Do not force the curve to be monotonic. Regressions and oscillations are meaningful signals used by drawdown, recovery, stagnation, and regret metrics. Results should store both progress_raw and progress_processed.

Postprocessing

The default pipeline is modular:

  1. Normalize the PRM output to absolute progress in [0, 1].
  2. Clip values to [0, 1].
  3. Replace isolated local outliers with the local median.
  4. Smooth with a weighted temporal window.
p'_t = 0.1*p_{t-2} + 0.2*p_{t-1} + 0.4*p_t + 0.2*p_{t+1} + 0.1*p_{t+2}

This default is meant to reduce isolated spikes and saw-tooth artifacts without hiding real regressions. It is especially useful when a human can see a clear upward trend but local pair judgments alternate between progress and no progress.

Disable postprocessing when you are debugging raw PRM behavior:

prm-judge eval --manifest cases.jsonl --outlier-method none --smoothing none

Change the smoothing weights from the CLI:

prm-judge eval --manifest cases.jsonl --smoothing-weights 0.05,0.15,0.6,0.15,0.05

The weight list should have odd length, non-negative values, and a positive sum. It is normalized automatically. For deeper customization, edit eval/prm_judge/curves.py, especially parse_smoothing_weights, weighted_smooth, and PostprocessConfig.smoothing_weights.

Metrics

Metrics are computed from the processed progress curve unless otherwise noted. Use raw curves for debugging and processed curves for the default evaluation report.

Case-Level Metrics

Metric Interpretation
MaxP Maximum progress reached. Useful for failed rollouts that still got close to success.
M25, M50, M75 Milestone indicators showing whether the rollout crossed progress thresholds.
SR Success indicator at the success threshold, default 0.99.
PPL Progress-weighted path length anchored at the best reached progress.
CRA Cumulative regret after regressions. Compare it under similar success or progress conditions.
Stag Fraction of near-stagnant transitions. Read it together with video length and data collection style.
MDD Maximum progress drawdown.
Rec Recovered progress after the deepest drawdown trough.
RBS Robustness score, approximately (Rec + epsilon) / (MDD + epsilon), clipped to 1.

Group-Level v1.5 Metrics

  • MaxP-F: average MaxP among failed trajectories.
  • SQS: success quality score for successful trajectories.
  • FNS: failure near-success score.
  • RBS: mean robustness after the largest drawdown.
  • OAS: overall auditing score combining success, near-success, and robustness signals.
Some diagnostics are conditional. For example, CRA is most informative when comparing rollouts that both achieved similar progress. A completely stagnant curve may have low regret but still be a poor rollout.

Custom PRM

The toolkit separates two responsibilities: a PRM adapter produces a progress curve, and the metric stack computes diagnostics from the standardized curve. This makes it possible to use PRMs beyond Dopamine.

Adapter checklist

  • Read EvalCase fields such as videos, task text, goal image, and metadata.
  • Run your PRM and return a progress sequence aligned to sampled frames or video time.
  • Declare whether the output is absolute progress, step-wise deltas, or another score.
  • Declare whether the scale is [0, 1], [0, 100], or uncalibrated.
  • Convert to absolute progress in [0, 1] before metrics.
from prm_judge.prm.base import BasePRMAdapter
from prm_judge.schema import EvalCase, ProgressTrace

class MyAdapter(BasePRMAdapter):
    def predict(self, case: EvalCase, output_dir):
        scores = run_my_model(case.videos, case.task, case.goal_image)
        return ProgressTrace(
            case_id=case.case_id,
            progress=scores,
            source_scale="0_1",
            source_type="absolute",
        )

Register the backend in eval/prm_judge/cli.py and document the model's output scale, semantics, required views, goal-image behavior, and runtime requirements.

RoboMeter as an example

RoboMeter is integrated as a sequence-style backend through an HTTP adapter. It reads outputs_progress.progress_pred as an absolute progress curve in [0, 1]. Because it reasons over a longer sequence, it is a useful comparison point for repeated-motion cases where local pairwise judging is not enough.

Recipes

Batch evaluate many videos without moving files

MANIFEST=/data/my_rollouts/cases.jsonl \
PRM_PATH=/models/Robo-Dopamine-GRM-8B-Pro \
OUT_DIR=eval/results/my_rollouts \
bash eval/run_eval.sh

Use all four local GPUs

GPUS=0,1,2,3 MANIFEST=cases.jsonl PRM_PATH=/path/to/prm bash eval/run_eval.sh

Run a denser curve for short manipulation tasks

FRAME_INTERVAL=36 MANIFEST=cases.jsonl PRM_PATH=/path/to/prm bash eval/run_eval.sh

Use benchmark labels for success grouping

prm-judge eval --manifest cases.jsonl --success-source label --visualize

Compare raw and processed progress

prm-judge eval --manifest cases.jsonl --outlier-method none --smoothing none --visualize
prm-judge eval --manifest cases.jsonl --visualize
